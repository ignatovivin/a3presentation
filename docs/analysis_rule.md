# Analysis Rule

## Обязательное стартовое правило для каждой задачи

Каждая новая задача в этом проекте должна начинаться с анализа проектных правил из папки `docs`.
Минимальный стартовый набор:

- [analysis_rule.md](analysis_rule.md)
- [document_class_matrix.md](document_class_matrix.md)
- [quality-contracts.md](quality-contracts.md)
- [system_backlog.md](system_backlog.md)
- [project_overview.md](project_overview.md)

Если задача затрагивает frontend, visual checks или chart preview, дополнительно нужно читать:

- [frontend-visual-contracts.md](frontend-visual-contracts.md)

Если задача выглядит простой технической командой, например показать `git status`, это правило можно не разворачивать в полный анализ.
Во всех остальных случаях сначала нужно сверить запрос с этими rule-docs, а уже потом читать код, менять файлы или делать выводы.

## Обязательный порядок разбора и исправления проблем

Для задач, связанных с генерацией, анализом и исправлением PowerPoint и DOCX, рабочий порядок такой:

1. Сначала опираться на внешнюю техническую документацию и спецификации:
   - PowerPoint Open XML
   - python-pptx
   - python-docx
2. Затем сопоставлять эти ограничения и возможности с внутренним кодом проекта.
3. После этого проверять, что решение синхронно работает в нашем пайплайне:
   - extraction
   - planning
   - generation
   - audit
4. Если документации недостаточно, разрешено добирать решение из интернета, предпочтительно из первичных технических источников.

## Практическое правило

Нельзя опираться только на эвристику или только на внутренний код.
Нужно комбинировать:

- внешние docs Open XML / python-pptx / python-docx
- внутреннюю реализацию проекта
- фактическую проверку результата на реальных `.docx` и `.pptx`

Дополнительно:

- проверка не должна опираться на один документ
- нужно сверяться с [document_class_matrix.md](document_class_matrix.md)
- любые значимые правки planner/generator/audit нужно оценивать по классам документов, а не по одному кейсу
- нельзя делать точечные фиксы под один документ, один слайд, один шаблон или один частный пример
- проблему нужно доводить до уровня общего механизма пайплайна: extractor, normalizer, planner, generator, audit или template-aware policy
- если найден баг на одном кейсе, он используется как индикатор класса проблем, а не как основание для локальной подгонки

## Цель

Добиваться того, чтобы решения были:

- технически корректны относительно формата `.pptx` / `.docx`
- совместимы с ограничениями используемых библиотек
- согласованы с архитектурой текущего проекта
- подтверждены реальным результатом генерации и анализа

## Обязательная документация по значениям на графиках

Для любых задач, где затрагиваются подписи значений, шкалы осей, `numFmt`, compact formats, `%`, `₽`,
data labels или mixed-unit charts, нужно опираться на следующий набор документации как на обязательную базу:

1. `python-pptx` chart tick labels:
   `https://python-pptx.readthedocs.io/en/stable/dev/analysis/cht-tick-labels.html`
2. `python-pptx` chart data labels:
   `https://python-pptx.readthedocs.io/en/latest/dev/analysis/cht-data-labels.html`
3. `python-pptx` chart API:
   `https://python-pptx.readthedocs.io/en/stable/api/chart.html`
4. `python-pptx` chart-data API:
   `https://python-pptx.readthedocs.io/en/latest/api/chart-data.html`
5. Open XML chart axis / `c:numFmt`:
   `https://learn.microsoft.com/en-us/dotnet/api/documentformat.openxml.drawing.charts.valueaxis.numberingformat?view=openxml-3.0.1`
6. Open XML chart structure examples:
   `https://learn.microsoft.com/en-us/office/open-xml/spreadsheet/how-to-insert-a-chart-into-a-spreadsheet`

### Рабочие правила, которые считаются обязательными

1. Формат подписей оси задаётся через `tick_labels.number_format`, а не через data labels.
2. Формат data labels и формат value-axis labels это разные контракты, их нельзя смешивать.
3. Если `number_format_is_linked` или `sourceLinked=1`, PowerPoint может тянуть формат из embedded workbook;
   при явном runtime-контракте формат нужно задавать как unlink'ed.
4. Процентный формат допустим только на оси, где все series процентные.
5. Mixed-unit график `number/RUB + %` не должен рендериться как single-axis chart с общей левой осью в `%`.
6. Для безопасного mixed-unit сценария нужно использовать `combo` и secondary value axis.
7. Compact number formats вроде `млн`, `млрд`, `₽`, `%` нужно назначать отдельно для каждой оси по фактическому набору series на этой оси.
8. Если серия процентов определяется не по ячейкам, а по semantic header вроде `Доля`, `Маржа`, `Процент`, это всё равно должно учитываться при выборе axis semantics.

### Операционное правило

Перед любым изменением логики chart formatting нужно:

1. свериться с этой документацией,
2. проверить текущий `chart_render_contract`,
3. проверить `TableChartAnalyzer -> planner -> generator -> deck_audit`,
4. после правки сделать свежую генерацию `.pptx` и проверить реальный chart XML/runtime artifact.

## Рабочее правило применения

Для решения проблем в проекте нужно всегда придерживаться такой последовательности:

0. Открыть и применить проектные документы с правилами, без ожидания отдельного напоминания от пользователя.
1. Проверить релевантные внешние технические документации.
2. Сопоставить их с ограничениями используемых библиотек.
3. Сверить выводы с внутренним кодом проекта.
4. Проверить решение на реальных входных и выходных артефактах.
5. После любой значимой правки генерации, планировщика, аудита, layout/capacity или template-aware логики обязательно запустить новую генерацию `.pptx` и проверить свежий результат, а не только ранее созданный файл или unit tests.
6. Если остаются пробелы, добрать решение из интернета, предпочтительно из первичных технических источников.

Нельзя:

- начинать значимый анализ или реализацию без обращения к проектным rule-docs, если только задача явно не является простой технической командой
- делать выводы только по внутреннему коду без сверки с документацией
- исправлять `.pptx` / `.docx` проблемы только эвристически, если вопрос затрагивает формат или библиотечные ограничения
- использовать случайные советы из интернета как основной источник без проверки на документации и на нашем коде
- лечить визуальный или структурный дефект точечно под один файл, если не доказано, что правка улучшает общий класс документов
- считать текущий шаблон, текущий документ или текущий пользовательский кейс константой для архитектурных решений

## Дополнительное правило глобальности решений

Любая найденная проблема должна формулироваться как класс проблем, а не как частный инцидент.

Рабочий стандарт такой:

1. Найти конкретный сбой на реальном кейсе.
2. Определить, какой общий механизм его порождает.
3. Исправить именно механизм, а не симптом на одном слайде или в одном шаблоне.
4. Закрепить это тестами и проверкой на других документах, шаблонах или классах документов.
5. Обязательно сгенерировать новый `.pptx` после правки и проверить в нём фактический результат: текст, размеры шрифтов, геометрию, порядок контента или другой исправляемый инвариант.

Это означает:

- проблемный файл используется как regression-case, но не как единственная цель оптимизации
- текущий шаблон не считается фиксированной константой
- решения должны быть совместимы с будущими пользовательскими и корпоративными шаблонами

## Дополнительное правило поиска соседних проявлений

Если найден один конкретный дефект, этого недостаточно для принятия правки как глобальной.

Рабочий стандарт такой:

1. Найти конкретное проявление проблемы.
2. Определить механизм, который его породил.
3. Проверить, нет ли того же механизма в соседних компонентах, layout'ах, типах слайдов, путях рендера или проверках.
4. Исправить весь класс проявлений, который безопасно подтверждается анализом, а не только первый найденный случай.
5. После правки обязательно сделать проверки: targeted regression tests, document-class checks, quality contracts, full suite и проверку на реальных артефактах.
6. Проверка на реальных артефактах означает именно новую генерацию после внесённых изменений; анализ старого `.pptx` допускается только для диагностики причины, но не считается подтверждением исправления.

Это означает:

- нельзя останавливаться на частичном "локально стало лучше", если есть основания ожидать те же симптомы в других связанных местах
- если проблема найдена в `planner`, нужно проверить соседние проявления в `generator`, `deck_audit`, template-aware логике и связанных UI/preview слоях, когда это применимо
- если проблема найдена в одном типе слайда или layout, нужно проверить родственные layout'ы и похожие сценарии
- внедрённая правка считается завершённой только после обязательной верификации на нескольких уровнях, а не после одного ручного кейса
- для генерационных задач обязательна свежая генерация `.pptx` после правки; без неё результат считается непроверенным

## Обязательная проверка смежных компонентов после изменений

После любой правки нужно считать, что она может повлиять на соседние компоненты пайплайна, даже если изменение выглядит локальным.

Рабочий стандарт такой:

1. Определить impacted components: какой слой изменён и какие слои зависят от его контракта.
2. Проверить прямой компонент целевым тестом или ручной runtime-проверкой.
3. Проверить смежные компоненты, которые читают или формируют тот же контракт.
4. Если изменение затрагивает генерационный путь, проверить цепочку `extractor -> planner -> generator -> deck_audit -> frontend preview/API`, выбирая релевантные звенья для конкретной задачи.
5. Если изменение затрагивает frontend preview или выбор пользователя, проверить backend/API contract, потому что preview и generator должны получать одинаковый `ChartSpec`/`PresentationPlan`.
6. Если изменение затрагивает backend analyzer/planner/generator, проверить frontend-поведение, которое зависит от этих данных: options, выбранные режимы, preview, download flow.
7. В итоговом ответе кратко указать, какие смежные компоненты были проверены и каким способом.

Это означает:

- нельзя считать правку завершённой только потому, что прошёл тест изменённого файла
- нужно проверять не только "сломался ли изменённый код", но и "не изменился ли контракт для соседей"
- при изменении данных, которые уходят в UI, нужно проверять UI/preview или хотя бы API-ответ, который этот UI потребляет
- при изменении UI, который формирует запрос на генерацию, нужно проверять, что backend принимает этот запрос и итоговый `.pptx` соответствует выбору пользователя
- если рядом есть известная связка компонентов, например `TableChartAnalyzer -> App chart controls -> ChartPreview -> PptxGenerator -> deck_audit`, она проверяется как единая цепочка, а не как независимые файлы

## Операционное правило автономного продолжения

Если следующий шаг логически вытекает из текущей задачи, безопасен и не требует отдельного продуктового выбора от пользователя, его нужно выполнять автоматически без дополнительного вопроса "продолжать или нет".

Рабочий стандарт такой:

1. Составить план.
2. Начать выполнять его сразу.
3. Доводить задачу до логически завершенного результата в рамках текущего запроса.
4. Спрашивать пользователя только там, где без этого нельзя выбрать направление, есть конфликт вариантов или требуется внешнее решение.

Это означает:

- не останавливаться после анализа, если можно сразу перейти к реализации
- не останавливаться после частичной правки, если можно безопасно продолжить следующий логичный этап
- не запрашивать лишнее подтверждение на продолжение внутри уже согласованной задачи
- использовать вопросы только для реальных развилок, а не как замену продвижению работы
